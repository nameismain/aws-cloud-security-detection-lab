# 탐지 기준 표준

## 개요
이 문서는 프로젝트에서 사용하는 탐지 기준의 공통 원칙을 정리합니다.

각 시나리오는 서로 다른 AWS 리소스와 로그를 다루지만, 탐지 로직은 일관된 기준으로 작성되어야 합니다.

## 탐지 기준 작성 원칙
탐지 기준은 다음 정보를 포함해야 합니다.

| 항목 | 설명 |
|---|---|
| 탐지 대상 | 어떤 리소스 또는 로그를 분석하는지 |
| 위험 조건 | 어떤 상태를 위험으로 판단하는지 |
| 기준 필드 | 판단에 사용하는 주요 필드 |
| 임계값 | 몇 회 이상 또는 어떤 조건 이상인지 |
| 시간 조건 | 특정 시간 윈도우가 필요한지 |
| 위험도 | Low, Medium, High 중 하나 |
| 오탐 가능성 | 정상 상황에서 탐지될 수 있는 경우 |
| 개선 권고 | 위험을 줄이기 위한 조치 |

## 위험도 기준

### High
즉시 보안 위험으로 이어질 수 있거나, 외부 공격자에게 직접적인 접근 가능성을 제공하는 경우입니다.

    예시:
        - SSH 포트가 0.0.0.0/0으로 공개
        - S3 버킷이 Public으로 공개
        - IAM에 AdministratorAccess 부여
        - CloudTrail StopLogging 발생

### Medium
조건에 따라 위험해질 수 있으며, 추가 분석이 필요한 경우입니다.

    예시:
        - 관리 포트가 넓은 사설 대역에 공개
        - IAM Policy에 Resource "*" 사용
        - 특정 버킷에 외부 계정 접근 허용
        - 비정상적인 API 호출 증가

### Low
즉각적인 공격 가능성은 낮지만 보안 개선이 필요한 경우입니다.

    예시:
        - 사용하지 않는 Access Key 존재
        - 오래된 IAM User 존재
        - 태그 누락으로 관리 주체 불명확
        - 로깅은 활성화되어 있으나 보존 정책 미흡

## 탐지 결과 형식
탐지 결과는 가능한 한 다음 형식으로 출력합니다.

    [위험도] 탐지명
    Resource: 영향을 받는 리소스
    Condition: 탐지된 조건
    Risk: 예상 위험
    Recommendation: 개선 권고

```
예시:
    [HIGH] Public SSH Exposure Detected
    Resource: sg-xxxxxxxx
    Condition: TCP 22 is open to 0.0.0.0/0
    Risk: Internet-facing SSH may allow brute force attempts.
    Recommendation: Restrict SSH access to trusted IP ranges.
```

## 오탐 검토 기준
탐지 결과를 작성할 때는 항상 오탐 가능성을 함께 검토합니다.

오탐 검토 질문은 다음과 같습니다.

    * 이 설정이 의도된 테스트 환경일 수 있는가?
    * 운영상 필요한 예외일 수 있는가?
    * 특정 신뢰 IP 또는 계정에 한정된 접근인가?
    * 단기적으로만 존재하는 임시 설정인가?
    * 추가 로그를 확인하면 정상 행위로 볼 수 있는가?

## 대응 원칙
초기 프로젝트에서는 실제 차단이나 삭제 같은 자동 대응을 수행하지 않습니다.

대신 Dry Run 방식으로 다음 정보를 출력합니다.

    * 어떤 리소스가 위험한지
    * 왜 위험한지
    * 어떤 조치가 필요한지
    * 실제 변경 전에 무엇을 확인해야 하는지

자동 대응은 충분한 검증 이후에만 고려합니다.
